# 提案: 工作流可观测性增强

## Why

当前 `scripts/workflow.py` 工作流虽然已有基本的步骤输出，但缺乏**中间 AI 处理结果的可观测性**。这导致以下问题：

1. **调试困难**: 用户无法看到中间步骤的具体数据流
2. **系统行为不透明**: 不清楚 AI 模型收到了什么样的提示词和数据
3. **数据质量验证困难**: 无法验证 ASR 转写质量、题库加载是否正确等

## 背景

当前 `scripts/workflow.py` 工作流虽然已有基本的步骤输出，但缺乏**中间 AI 处理结果的可观测性**。用户无法看到：
- 音频转写的 ASR 原始输出
- AI 评测模型收到的提示词和上下文
- AI 评测模型返回的中间推理过程（如果有）

这导致调试困难和对系统行为的不透明性。

## 目标

增强工作流的可观测性，让**每一步的 AI 结果都输出到控制台**，便于用户理解流程、调试问题和验证数据质量。

## 范围和相关规格

### 涉及的现有规格
- `evaluation-workflow`: 定义了工作流的核心功能

### 变更涉及的模块
1. `scripts/workflow.py` - 主工作流脚本
2. `scripts/qwen3.py` - 评测引擎（需要配合输出）
3. `scripts/captioner_qwen3.py` - 音频转写（需要配合输出）

## 主要设计决策

### 1. 输出格式
- **ASR 转写结果**: 原始输出 + 格式化展示（JSON 若可解析）
- **AI 提示词**: 展示完整的 system_prompt、qb_prompt、asr_prompt
- **评测结果**: JSON 格式原始输出

### 2. 控制台输出风格
- 使用分隔符区分各段落（已存在的 `"=" * 60` 模式）
- 添加清晰的标签说明各部分内容
- 保留原有的进度指示符（✓, ✅, ❌, 🎵, 🤖 等）
- 可选配置：增加 `--verbose` 或 `--debug` 标志以控制详细程度

### 3. 向后兼容性
- 默认行为：输出中间结果（满足可观测性需求）
- 可扩展：支持 `--quiet` 标志用于生产环境（仅输出最终结果）

## 预期变更

### Requirement 1: AI 转写结果可观测性
在完成音频转写后，输出 ASR 原始结果到控制台。

### Requirement 2: AI 评测提示词可观测性
在调用评测 AI 前，输出完整的三层提示词结构到控制台。

### Requirement 3: AI 评测结果可观测性
在完成评测后，输出最终评测结果 JSON 到控制台。

### Requirement 4: 可观测性控制选项
添加命令行标志以控制输出详细程度（可选扩展）。

## What Changes

### 修改文件：`scripts/workflow.py`

**变更内容：**
1. 在 `run_workflow()` 函数中的第2步（音频转写）完成后，添加 ASR 结果输出（~4行代码）
2. 在第3步（题库加载）完成后，添加题库摘要输出（~6行代码）
3. 在第4步（AI评测）执行前，添加三层提示词结构输出（~15行代码）
4. 在第4步完成后，添加评测结果 JSON 输出（~6行代码）

**总计新增代码：** ~31行（不含注释和空白）

**影响范围：**
- 仅修改 `run_workflow()` 函数体
- 函数签名和返回值保持不变
- 没有影响其他模块（qwen3.py、captioner_qwen3.py）
- 完全向后兼容

**关键设计：**
- 使用一致的分隔符和 emoji 标签进行格式化输出
- 对超过 500 字符的内容自动截断并显示"..."
- JSON 结果使用 `json.dumps(..., indent=2, ensure_ascii=False)` 美化输出
- 所有输出仅写入 stdout，不修改返回值

**验证清单：**
- [x] 语法通过 `python3 -m py_compile` 验证
- [x] 四个可观测性输出点按预期显示
- [x] 输出格式清晰可读
- [x] 原有功能不受影响
- [x] 模块独立性保持完整
