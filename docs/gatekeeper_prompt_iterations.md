# Gatekeeper Prompt 迭代总结

## 迭代历程

### v1.0.0 - 初始版本
**问题**:
- 逻辑简单，检测规则不够清晰
- 没有明确老师念题流程
- 缺少具体示例

### v1.1.0 - 明确念题流程
**改进**:
- 明确老师念题流程: question → 停顿 → answer
- 定义 ASR 中先出现的词 = question 的语言
- 增加翻译方向检测逻辑

**问题**:
- "英翻中"/"中翻英"表述混乱
- 容易误导模型判断方向

### v1.2.0 - 统一表述
**改进**:
- 统一使用"中→英"/"英→中"箭头表述
- 明确题库方向与 ASR 模式的匹配逻辑
- 提供清晰的对比表格

**问题**:
- 只看第一个词对，不够稳健
- 学生漏录时前几题可能不准确
- 保守策略（不确定→PASS）可能漏检

### v1.3.0 - 综合分析 + 严格策略 ✓

**核心改进**:

1. **综合分析多个词对**
   ```
   旧: 只看前200字符的第一个词对
   新: 分析全部内容（或至少500字符）的所有词对
   ```

2. **统计主导模式**
   ```python
   # 伪代码逻辑
   中→英_count = 0
   英→中_count = 0

   for 词对 in ASR转写:
       if 中文在前, 英文在后:
           中→英_count += 1
       elif 英文在前, 中文在后:
           英→中_count += 1

   if 中→英_count > 英→中_count × 1.5:
       ASR模式 = "中→英"
   elif 英→中_count > 中→英_count × 1.5:
       ASR模式 = "英→中"
   else:
       ASR模式 = "不清晰"
   ```

3. **严格策略**
   ```
   旧: 不确定时 → PASS（保守，避免误杀）
   新: 不确定时 → FAIL（严格，需人工检查）
   ```

**理由**:
- 学生经常漏录，导致前几题不准确
- 需要看更多词对才能准确判断
- 宁可误杀（让人工检查），不可漏检（浪费 LLM 资源）

## 关键示例

### 正确案例 (PASS)

**Cathy** (Zoe41900_2025-09-08):
```
题库: question="不" answer="not" (中翻英)
ASR: "不，not。not。双倍的，双的。呃，double double。一半，半。half..."

分析:
- "不 not"(中→英) ✓
- "双倍的 double"(中→英) ✓
- "一半 half"(中→英) ✓

统计: 中→英(多个), 英→中(0)
主导: 中→英 占绝对优势
判断: 中翻英 = 中→英
→ PASS ✓
```

### 错误案例 (FAIL)

**Benjamin** (Abby61000_2025-11-05):
```
题库: question="小孩" answer="kid" (中翻英)
ASR: "字表二十七英翻中。kid kid 呃小孩。小孩。pupil 小学生，瞳孔..."

分析:
- "kid 小孩"(英→中) ✗
- "pupil 小学生"(英→中) ✗
- "baby 婴儿"(英→中) ✗

统计: 中→英(0), 英→中(多个)
主导: 英→中 占绝对优势
判断: 中翻英 ≠ 英→中
→ FAIL - WRONG_QUESTIONBANK ✓
```

### 混乱案例 (FAIL - 严格策略)

**Jeffery** (Abby61000_2025-11-05):
```
题库: question="小孩" answer="kid" (中翻英)
ASR: "二十七 in 班中。kid 小孩。now 小孩，pupil 小学生..."

分析:
- "kid 小孩"(英→中) - 清晰
- 其他词对被 "now"、"in" 等噪音词干扰

统计: 中→英(0), 英→中(1), 噪音多
主导: 无法确定（样本太少且有噪音）
判断: ASR模式 = 不清晰
→ FAIL - WRONG_QUESTIONBANK ✓（严格策略）
```

## 最终 Prompt 结构

### system.md
1. 背景知识
   - 老师念题流程
   - 关键推论（ASR词序 = question/answer 语言）
2. 检测问题类型
   - 题库错误（方向错误 + 词汇不匹配）
   - 音频异常
3. 判断步骤（3步）
   - 从题库判断方向
   - 从 ASR 提取模式（综合分析）
   - 对比是否一致（严格策略）
4. 详细示例（3个）
   - 正确示例
   - 错误示例
   - 不清晰示例

### user.md
1. 输入数据（题库 + ASR）
2. 处理指令（5步）
   - 步骤1: 判断题库翻译方向
   - 步骤2: 提取 ASR 实际模式（**综合分析多个词对**）
   - 步骤3: 对比题库方向与 ASR 模式（**严格策略**）
   - 步骤4: 检查其他异常
   - 步骤5: 输出判定结果
3. 输出要求（严格 JSON）

## 关键改进点总结

| 维度 | v1.0-1.2 | v1.3.0 |
|------|---------|--------|
| 分析范围 | 前200字符，第一个词对 | 全部内容，所有词对 |
| 判断方式 | 单一词对 | 统计主导模式 |
| 漏录处理 | 容易误判 | 综合分析，更稳健 |
| 不确定策略 | PASS（保守） | FAIL（严格） |
| 适用场景 | 理想情况 | 实际场景（含漏录） |

## 使用建议

1. **正常使用**: 直接运行，FAIL 时人工检查题库选择
2. **调试模式**: 使用 `--verbose` 查看详细判断过程
3. **批量处理**: FAIL 的学生单独处理，不影响 PASS 的学生
4. **持续改进**: 收集误判案例，继续优化 prompt
